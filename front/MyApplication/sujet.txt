Sujet : Application Bancaire
Ce projet est une application bancaire dont l'objectif est de reproduire le fonctionnement d'une banque en ligne, permettant la gestion des utilisateurs, la détention de multiples comptes bancaires, l'exécution de virements et la gestion d'offres commerciales.



Liste des API
Account-API
Bank-Account-API
Operation-API
Commercial-offer-api
Gateway

Architecture






Liste des Utilisateurs
Visiteur
Visualisation des offres.
Client
Accès à ses compte, virement.
Administrateur
Blocage d’un compte bancaire, annulation d’un virement, injection d’argent.





Screens
Visiteur
Accueil : affiche les boutons connexion, inscription, offres
Offre : Affiche les différentes offres disponibles.
Connexion : Affiche les champs de saisie pour se connecter (code de compte et mot de passe).
Inscription : Différents champs de saisie pour pouvoir faire une inscription complète.
Client
Liste des Comptes possédé : Affiche la liste des comptes que l’on possède sans les détails juste nom et solde.
Accueil Client : Affiche le solde du compte courant.
Détails Compte : affiche les détails d’un compte avec la liste des opérations, le découvert autorisé etc …
Bénéficiaire : CRUD des bénéficiaires.
Virements : Affiche les options obligatoire pour effectuer un virement (suite d’écran)
Admin
Admin Panel Gestion des Comptes : Permet de gérer des comptes (RUD + state)
Admin Panel gestion des Comptes Bancaire: Permet de gérer des comptes Banquaire (RUD + state)
Admin Panel gestion des Opérations: Permet de gérer des Opérations (CR + state + annulation)
Admin Panel gestion des Offres: Permet de gérer des Offres (CRUD + state)
Admin Panel gestion type de compte bancaire : Permet de gérer les types de compte bancaire (CRUD)

Acteur : Visiteur
Use Case 1 : Inscription d’un nouveau client
Acteur : Visiteur
But : Créer Compte
Pré-condition : Aucune
Scénario Nominal :
Le visiteur demande la création d’un compte.
Le système demande les informations personnelles du visiteur.
Le visiteur saisit ses informations personnelles.
Le visiteur définit un mot de passe.
Le système vérifie la conformité des informations fournies.
Le système crée le profil client.
Le système crée un compte avec l’état INACTIF.
Le système confirme la création du compte.
Scénarios d’Échec :
3a – Champ obligatoire manquant ou invalide
Le système détecte une information manquante ou incorrecte.
Le système affiche un message d’erreur.
Le compte n’est pas créé.
Use Case 2 : Connexion
Acteur : Visiteur
But : Accéder à son espace personnel
Pré-condition : Le compte existe
Scénario Nominal :
Le visiteur saisit son identifiant et son mot de passe.
Le système vérifie les informations d’authentification.
Le système ouvre la session du visiteur.
Scénarios Alternatifs :
2a – Identifiants incorrects
Le système rejette l’authentification.
Le système affiche un message d’erreur.
Le visiteur saisit à nouveau son identifiant et son mot de passe.
Le système vérifie les informations d’authentification.
Le système ouvre la session du visiteur.
Scénarios d’Échec :
2a – Identifiants incorrects
Le système rejette l’authentification.
Le système affiche un message d’erreur.
La session n’est pas ouverte.
2c – Compte inactif
Le système détecte que le compte est inactif.
Le système refuse la connexion.
Un message d’information est affiché.
